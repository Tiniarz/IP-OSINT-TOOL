## Technical Overview

The application functions as a static execution broker. Instead of importing submodules into the active memory space, it leverages decoupled subprocess mechanics. This structural isolation ensures that variables, dependency versions, and thread loops inside individual utilities do not pollute or crash the primary routing panel.

### Explanation

1. **Terminal Discovery**: The engine dynamically queries the active terminal boundary dimensions to adjust column distributions for the streaming simulation.
2. **Procedural Rendering**: The main control panel loads content sequentially using fixed timing intervals to maintain uniform line spacing during initialization.
3. **Execution Routing**: Input choices translate to dynamic relative path bindings, resolving to the localized module storage container.
4. **Handoff Mode**: The active interface temporarily yields standard output and input streams to the called utility, resuming automatic polling immediately upon execution termination.

## Directory Architecture

The framework relies on a rigid local directory tree. Sub-scripts must be placed within the dedicated execution package folder relative to the orchestrator file path.
